# Soupify Format Instructions (v0.2.0)

You will receive a "soup" file — a markdown file bundling multiple source files. Parse it, perform the requested work, and return results in soup format so the user can restore files locally.

## Overview

**Workflow:**
1. Receive a soupified (`.soup` or `.md`) file containing multiple source files.
2. Parse block headers to reconstruct the original files.
3. Perform the requested tasks (analysis, modifications, debugging, etc.).
4. Resoupify only modified/new files back into a soup file.
5. Return the soup file so the user can desoupify to restore files locally.

## Document Structure

Zero or more **meta blocks** (always first) followed by zero or more **file blocks**. Blocks concatenate directly — no blank lines between them. A trailing newline after the last block is tolerated.

## File Block Header

```
#SOUP "<json_escaped_abs_path>" [#SOUP_PARTIAL_LINES <s>-<e>] #SOUPED_LINES <N> #SOUP_TRAILING_NEWLINE <0|1> [#SOUP_BASE_SHA <hex>] [#SOUP_READONLY <true|false>]
```

- **Path** — Original absolute path, JSON-encoded. Preserve exactly when resoupifying.
- **#SOUP_PARTIAL_LINES** *(optional)* — Replaces only lines `s`–`e` (1-indexed, inclusive) of the existing file.
- **#SOUPED_LINES** — Exact count of content lines following the header.
- **#SOUP_TRAILING_NEWLINE** — `1` = file ends with `\n`, `0` = it does not.
- **#SOUP_BASE_SHA** *(optional)* — blake3 hash of the file at soupify time. When returning a PARTIAL block, carry this forward so the host can detect drift. If the on-disk file changed since soupification, desoupify refuses the partial.
- **#SOUP_READONLY** *(optional)* — `true` means this block is context-only. Read it, honor it, but NEVER modify it and NEVER emit it in resoupified output.

**Examples:**
```
#SOUP "/home/user/src/Button.tsx" #SOUPED_LINES 42 #SOUP_TRAILING_NEWLINE 1
#SOUP "/home/user/src/main.rs" #SOUP_PARTIAL_LINES 5-10 #SOUPED_LINES 6 #SOUP_TRAILING_NEWLINE 1
```

## Meta Block Header

```
#SOUP_META "<label>" #SOUP_META_KIND <kind> #SOUP_META_FORMAT <fmt> #SOUP_META_LINES <N> #SOUP_META_READONLY <true|false>
```

Generated by `soupify -g`. Always appears before file blocks. Parse `N` lines of content then **skip — do not write to disk**. Do not emit meta blocks yourself.

**Example:** `#SOUP_META "repo-graph" #SOUP_META_KIND codegraph #SOUP_META_FORMAT repomap #SOUP_META_LINES 120 #SOUP_META_READONLY true`

## Parsing a Soup File (Desoupify)

1. Scan for `#SOUP_META` or `#SOUP` header lines.
2. **Meta block:** extract `#SOUP_META_LINES N`, consume exactly N lines, discard.
3. **File block:** JSON-decode path, read optional partial range, `N`, trailing-newline flag; consume exactly N content lines.
4. Reconstruct each file:
   - **Full block:** join lines with `\n`, append `\n` if flag is `1`. Create parent dirs. Write to disk.
   - **Partial block:** read target file as it currently exists on disk (reflecting any prior partial edits to it). Replace lines `s`–`e` (1-indexed) with block content. Write merged result back.
5. Repeat until all blocks are processed.

**Notes:**
- Header regex: `^#SOUP ("(?:\\.|[^"\\])*")(?: #SOUP_PARTIAL_LINES ([0-9]+)-([0-9]+))? #SOUPED_LINES ([0-9]+) #SOUP_TRAILING_NEWLINE (0|1|true|false)$`
- Create parent directories automatically (mkdir -p equivalent).
- Partial blocks require the target file to exist and be long enough to cover `end_line`.
- Multiple partial blocks on the same file apply in document order; each sees the result of the previous.

## Outputting Edited Files (Resoupify)

Only emit files you modified or created. Two options per file:

### Full block — rewritten or new file
1. Read the complete file content.
2. Count lines: strip trailing `\n` if present, split on `\n`, count segments (empty → 0).
3. Determine whether content ends with `\n`.
4. Emit `#SOUP "<path>" #SOUPED_LINES <N> #SOUP_TRAILING_NEWLINE <0|1>` then all content lines.

### Partial block — localized change in a large file
1. Identify the line range changed (1-indexed, inclusive).
2. Count replacement lines with the same logic as above.
3. Emit `#SOUP "<path>" #SOUP_PARTIAL_LINES <s>-<e> #SOUPED_LINES <N> #SOUP_TRAILING_NEWLINE <0|1>` then replacement lines only.
4. Trailing-newline flag = the **whole file's** state after the edit, not just the snippet.
5. When emitting multiple partial blocks for the same file, line numbers in later blocks must account for line-count changes introduced by earlier partial blocks.

## Line Counting Rules

| Content | Lines | Trailing |
|---|---|---|
| *(empty)* | 0 | 0 |
| `hello` | 1 | 0 |
| `hello\n` | 1 | 1 |
| `a\nb\n` | 2 | 1 |
| `a\nb` | 2 | 0 |
| `one\n\nthree\n` | 3 | 1 |

## Examples

**Two-file soup:**
```
#SOUP "/tmp/src/main.rs" #SOUPED_LINES 3 #SOUP_TRAILING_NEWLINE 1
fn main() {
    println!("Hello, world!");
}
#SOUP "/tmp/Cargo.toml" #SOUPED_LINES 2 #SOUP_TRAILING_NEWLINE 0
[package]
name = "hello"
```

**With meta block** (from `soupify -g`) — meta content parsed then skipped, only `index.js` written:
```
#SOUP_META "repo-graph" #SOUP_META_KIND codegraph #SOUP_META_FORMAT repomap #SOUP_META_LINES 1 #SOUP_META_READONLY true
<graph data>
#SOUP "/tmp/index.js" #SOUPED_LINES 1 #SOUP_TRAILING_NEWLINE 0
console.log('hi');
```

**Partial edit** — replace lines 2–3 of a 4-line file that ends with `\n`:
```
#SOUP "/tmp/file.txt" #SOUP_PARTIAL_LINES 2-3 #SOUPED_LINES 2 #SOUP_TRAILING_NEWLINE 1
dos
tres
```

## Error Handling

- **Malformed header** — report the line number and problematic content.
- **Line count mismatch** — report expected vs. actual and the affected file path.
- **Invalid partial range** — report if `start > end`, either value is `0`, or `end` exceeds existing file length.
- **Missing metadata** — report which of `#SOUPED_LINES` or `#SOUP_TRAILING_NEWLINE` is absent and the affected path.

## Best Practices

- Validate soup block headers before consuming file content.
- Track modifications carefully so you can resoupify accurately.
- Line counts must be exact — off-by-one errors corrupt the soup file.
- Handle the trailing newline flag correctly; it is essential for exact file restoration.
- Empty files: `#SOUPED_LINES 0 #SOUP_TRAILING_NEWLINE 0` with no content lines after the header.
- Prefer partial blocks for large files with small changes to reduce output size.
- Add any newly created files to the resoupified output even if they weren't in the original soup.
- Do not emit `#SOUP_META` blocks — those are produced by the soupify CLI, not by AI.
- Do not emit `#SOUP_READONLY true` blocks — those are context-only; never resoupify them.
- Carry `#SOUP_BASE_SHA` forward unchanged when returning partial blocks for the same file.
- If the user requests specific files, only resoupify those files.

## Requesting Files

The repo-map meta block may reference files NOT present as editable #SOUP blocks. Those are context only. If you need the full contents of such a file to complete the task, do NOT infer or hallucinate them — request the file by emitting a line in your text response:

```
#SOUP_REQUEST "<absolute_path>" <short reason>
```

The user can then re-soup with that file included (e.g. `soupify --seed <path> ...`).
